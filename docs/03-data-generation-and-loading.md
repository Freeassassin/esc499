# Data Generation and Loading

This document describes how synthetic benchmark data is generated by the TPC-H and TPC-DS reference toolkits, how database schemas are defined per engine, and how data is loaded into each of the four target databases.

---

## 1. TPC-H Data Generation

### 1.1 Data Generator: dbgen

TPC-H data is generated by `dbgen`, the C reference implementation provided by the TPC-H specification. The source code resides in `TPC-H/dbgen/` and is compiled with a configurable `DATABASE` flag:

```bash
make -j1 DATABASE=$TPCH_DBGEN_DATABASE   # Default: ORACLE
```

The `DATABASE` flag affects only compilation flags (`-D$(DATABASE)` in CFLAGS), not the output data format—all dbgen outputs are identical regardless of the compilation target.

### 1.2 Generation Process

Data generation is orchestrated by `TPC-H/tools/generate_data.sh`:

1. **Locking**: Acquires an exclusive `flock(2)` lock on `data/.locks/sf_{scale}.lock` to prevent concurrent generation for the same scale factor
2. **Idempotency check**: If `data/{scale}/.done` exists, generation is skipped (unless `--force` is specified)
3. **Compilation**: Builds `dbgen` and `qgen` from C source
4. **Generation**: Executes `./dbgen -f -s {scale_factor}` to produce pipe-delimited `.tbl` files
5. **Validation**: Checks that `region.tbl` contains exactly 5 rows (TPC-H standard sanity check)
6. **Metadata**: Writes `metadata.txt` with row counts per table and generation timestamp
7. **Marker**: Creates `.done` marker file to signal successful completion

### 1.3 Output Schema

TPC-H produces **8 tables** with the following cardinalities at scale factor 1 (SF=1):

| Table | Rows at SF=1 | Scaling |
|:------|:-------------|:--------|
| `region` | 5 | Constant |
| `nation` | 25 | Constant |
| `supplier` | 10,000 | Linear with SF |
| `customer` | 150,000 | Linear with SF |
| `part` | 200,000 | Linear with SF |
| `partsupp` | 800,000 | Linear with SF |
| `orders` | 1,500,000 | Linear with SF |
| `lineitem` | ~6,000,000 | Linear with SF |

**Output format**: Pipe-delimited text files (`|` separator), one file per table, stored in `TPC-H/data/{scale}/`. Each line has a trailing pipe character that must be handled during loading.

### 1.4 Supported Scale Factors

The framework has been tested with scale factors: 0.01, 1, 10, 20, 50. The default benchmark configuration tests: 1, 5, 10, 20, 50, 100, 500, 1000, 3000, 5000.

## 2. TPC-DS Data Generation

### 2.1 Data Generator: dsdgen

TPC-DS data is generated by `dsdgen`, the C reference implementation provided by the TPC-DS specification. The source code and build system reside in `TPC-DS/tools/`.

### 2.2 Generation Process

Data generation is orchestrated by `TPC-DS/tools/generate_data.sh`:

1. **Locking**: Uses a lock file to prevent parallel generation for the same scale factor
2. **Idempotency check**: Checks for `.generated.json` marker file under `data/sf{scale}/`; if present and file count matches expected (26 files), generation is skipped
3. **Compilation**: `make -C tools OS=LINUX all` (compiled once per session)
4. **Index build**: `distcomp` builds the distribution index
5. **Generation**: `dsdgen -force y -rngseed {seed} -quiet y` with configurable seed (default: 100)
6. **Validation**: Counts `.dat` output files, expecting 26 (one per TPC-DS table)
7. **Metadata**: Saves `.generated.json` with scale, seed, timestamp, and file count

### 2.3 Output Schema

TPC-DS produces **25 tables** (with a 26th file for `dbgen_version`), including 7 fact tables and 18 dimension tables:

**Fact tables**: `catalog_returns`, `catalog_sales`, `inventory`, `store_returns`, `store_sales`, `web_returns`, `web_sales`

**Dimension tables**: `call_center`, `catalog_page`, `customer`, `customer_address`, `customer_demographics`, `date_dim`, `household_demographics`, `income_band`, `item`, `promotion`, `reason`, `ship_mode`, `store`, `time_dim`, `warehouse`, `web_page`, `web_site`

**Output format**: Pipe-delimited (`.dat` files), no headers, empty fields represent NULL values. Stored in `TPC-DS/data/sf{scale}/`.

## 3. Query Generation

### 3.1 TPC-H Queries

TPC-H query generation uses `qgen`, the reference query generator compiled from `TPC-H/dbgen/`:

1. **Raw generation**: `DSS_QUERY=queries ./qgen -s {stream} {query_number}` for each of 22 queries
2. **Raw cache**: Output stored in `queries/raw/{engine}/{stream}/` for reuse
3. **Dialect transformation (Phase 1 — sed)**: Engine-specific regex substitutions from `tools/query_transforms/{engine}.sed` (e.g., interval syntax fixes: `interval '78' day (3)` → `interval '78 day'`)
4. **Dialect transformation (Phase 2 — Perl)**: Engine-specific Perl post-processing from `tools/query_transforms/{engine}.pl` (e.g., UDF syntax conversion, DDL replacement for StarRocks)
5. **Output**: Final queries stored in `TPC-H/queries/{engine}/{stream}/{1-22}.sql`

**Stream support**: Different stream numbers produce different parameter substitutions from the same query template, enabling multi-stream benchmarking with varied but deterministic query parameters.

### 3.2 TPC-DS Queries

TPC-DS query generation uses `dsqgen`, the reference query generator:

1. **Per-query generation**: For each of 99 queries, `dsqgen` is invoked with:
   ```
   dsqgen -input query{i}.tpl -dialect {engine} -directory templates
          -output_dir {out} -scale {sf} -streams {stream} -rngseed {seed}
   ```
2. **Template substitution**: `dsqgen` replaces `[PLACEHOLDERS]` in `.tpl` files with scale-factor-specific values
3. **Output**: Individual files `query{1-99}.sql` stored in `TPC-DS/queries/{engine}/sf{scale}/stream{stream}/`
4. **Metadata**: `.generated.json` marker in query output directory

**Query templates**: 99 base templates (`query1.tpl` through `query99.tpl`) plus 14 variant templates (`query5a.tpl`, `query14a.tpl`, etc.) stored in `TPC-DS/query_templates/`. Variant templates provide alternate formulations of the same analytical question for workload diversity testing.

### 3.3 SQL Dialect Transformations

Each engine requires different SQL syntax for identical operations. The transformation pipeline handles these differences:

| Feature | PostgreSQL | DuckDB | CedarDB | StarRocks |
|:--------|:-----------|:-------|:--------|:----------|
| Date arithmetic | `+ INTERVAL '78 days'` | `+ INTERVAL 78 DAYS` | `+ INTERVAL '78 days'` | `+ interval 78 day` |
| String functions | Standard SQL | Standard SQL | Standard SQL | `substring(x, N, M)` instead of `SUBSTRING(x FROM N FOR M)` |
| Date extraction | `EXTRACT(YEAR FROM x)` | Standard | Standard | `YEAR(x)` instead of `EXTRACT(YEAR FROM x)` |
| Division safety | `NULLIF()` wrap | Native handling | `NULLIF()` wrap | Direct |
| Row limit | Standard | Standard | Standard | `limit -1` → `limit N` conversion |
| DDL in queries | Allowed | Allowed | Allowed | Replaced with `SELECT 1 AS ddl_probe` |

For TPC-DS, additional runtime normalisation is applied by `pipeline_common.py:normalize_sql()`, which handles:

- Date interval syntax conversion per engine
- Division-by-zero protection via `NULLIF()` wrapping (PostgreSQL/CedarDB)
- PostgreSQL `lochierarchy` ORDER BY fix using `GROUPING()` expressions
- StarRocks-specific DML/DDL detection and probing (replacing with dummy `SELECT 1`)
- Reserved word conflicts (`at` → `at_tbl`, `c_last_review_date_sk` → `c_last_review_date`)
- Alias conflicts (`returns` → `returns_amt`)
- Subquery aliasing for StarRocks parser compatibility
- `CURRENT_TIMESTAMP` type cast conversion for StarRocks

## 4. Database Schema Design

### 4.1 TPC-H Schemas

Each engine has a dedicated DDL file in `TPC-H/{engine}/ddl.sql`:

**PostgreSQL** (`TPC-H/postgresql/ddl.sql`):
- Full relational model with PRIMARY KEYs on all tables
- 8 FOREIGN KEY constraints modelling the TPC-H relationships
- 3 secondary indexes: `idx_orders_custkey`, `idx_lineitem_orderkey`, `idx_lineitem_part_supp`
- Types: `NUMERIC(15,2)` for decimals, `CHAR(N)` / `VARCHAR(N)` for strings

**DuckDB** (`TPC-H/duckdb/ddl.sql`):
- Minimal schema: `INTEGER`, `BIGINT`, `DECIMAL(15,2)`, `VARCHAR` (no length limits)
- No PRIMARY KEYs, FOREIGN KEYs, or indexes (DuckDB treats constraints as advisory hints)

**CedarDB** (`TPC-H/cedardb/ddl.sql`):
- PostgreSQL-compatible: `VARCHAR(N)` with size limits, same FOREIGN KEY relationships
- Same 3 indexes as PostgreSQL
- No PRIMARY KEY constraints in table definitions

**StarRocks** (`TPC-H/starrocks/ddl.sql`):
- OLAP-specific table definitions using `ENGINE=OLAP` with `DUPLICATE KEY` strategy
- Hash distribution: `DISTRIBUTED BY HASH({key}) BUCKETS {N}` (1 for small tables, 8 for large)
- Single replica: `PROPERTIES ("replication_num" = "1")`
- Creates dedicated database: `CREATE DATABASE IF NOT EXISTS tpch; USE tpch;`
- No constraints or indexes (handled by OLAP engine internally)

### 4.2 TPC-DS Schema

All engines share a common DDL defined in `TPC-DS/tools/tpcds.sql`, with engine-specific execution handled by `TPC-DS/tools/prepare_schema.py`:

**DuckDB**: Executes DDL directly against a new `.duckdb` file at `TPC-DS/logs/duckdb/sf{N}/tpcds_sf{N}.duckdb`. Sets `PRAGMA threads={N}` for parallel operations.

**PostgreSQL**: Drops all existing TPC-DS tables with CASCADE, then executes DDL with `autocommit=True`.

**CedarDB**: Same approach as PostgreSQL (psycopg connection on port 5433).

**StarRocks**: Requires additional setup:
1. Waits for FE availability (timeout: 180 seconds)
2. Verifies backend registration and health (`Alive=true`, `Status=OK`)
3. Drops and recreates the database
4. Converts the generic DDL via `parse_tpcds_schema()` and `create_table_sql()`:
   - OLAP engine tables with `DUPLICATE KEY` on first column
   - Hash distribution: 8 buckets for fact tables, 1 bucket for dimension tables
   - Custom type mapping (e.g., `TIME` → `VARCHAR`)
   - `replication_num = 1`

## 5. Data Loading Pipelines

### 5.1 TPC-H Data Loading

`TPC-H/tools/load_data.py` implements engine-specific loading with **FK dependency ordering** via phased loading:

| Phase | Tables | Strategy | Rationale |
|:------|:-------|:---------|:----------|
| 1 | `region` | Serial | Base dimension, no dependencies |
| 2 | `nation` | Serial | Depends on region |
| 3 | `supplier`, `customer`, `part` | Parallel (up to 4 workers) | Independent dimensions |
| 4 | `partsupp` | Serial | Depends on part + supplier |
| 5 | `orders` | Serial | Depends on customer |
| 6 | `lineitem` | Serial | Depends on orders + partsupp |

Parallel loading in Phase 3 uses `ThreadPoolExecutor` with configurable worker count (`TPCH_LOAD_JOBS`, default: 4).

**PostgreSQL loading**:
```
COPY {table} FROM STDIN WITH (FORMAT text, DELIMITER '|')
```
Pre-processing: Removes trailing pipe character from each line (`line[:-2] + "\n"` if line ends with `|\n`). Row counts validated after each table load.

**DuckDB loading**:
```sql
INSERT INTO {table} SELECT ... FROM read_csv('{file}',
    delim='|', header=false, auto_detect=false, sample_size=-1, ignore_errors=false)
```
Explicit column selection handles trailing pipe. Runs `ANALYZE` at the end to gather statistics. Output database: `TPC-H/duckdb/tpch_sf{scale}.duckdb`.

**CedarDB loading**: Same `COPY FROM STDIN` approach as PostgreSQL (psycopg on port 5433), with configurable parallel jobs (`TPCH_CEDAR_LOAD_JOBS`, default: 3).

**StarRocks loading**:
1. Verifies backend health via `SHOW BACKENDS` polling (60-second timeout)
2. Applies DDL statements individually via MySQL connection
3. For each table:
   a. Cleans file (removes trailing pipes into temporary file)
   b. Constructs HTTP Stream Load URL: `PUT /api/{database}/{table}/_stream_load`
   c. Sets headers: `column_separator: |`, `columns: {list}`, `label: tpch_{table}_{uuid}`, `null_format`, `strict_mode: false`, `timeout: 7200`
   d. Authorization: `Basic` auth with base64-encoded `user:password`
   e. Sends via `requests.put()` with 7,200-second timeout
   f. Handles HTTP 307/308 redirects (StarRocks FE may redirect to BE)
   g. Validates response JSON: `Status` must be `"Success"` or `"Publish Timeout"`

### 5.2 TPC-DS Data Loading

`TPC-DS/tools/load_data.py` implements similar engine-specific loading:

**DuckDB**: Opens database file, uses `COPY {table} FROM '{file}' (DELIMITER '|', HEADER FALSE, NULLSTR '', AUTO_DETECT FALSE)` for each `.dat` file.

**PostgreSQL**: Iterates `.dat` files in sorted order, uses `COPY FROM STDIN WITH (FORMAT CSV, DELIMITER '|', HEADER FALSE, NULL '')`. Pre-processing: removes trailing pipe if present.

**CedarDB**: Same COPY approach as PostgreSQL. Additional pre-processing: `clean_data_file()` normalises empty fields to `\N` (PostgreSQL NULL representation) and removes trailing pipes. Output via temporary normalised file.

**StarRocks**: Uses the same HTTP Stream Load API as TPC-H loading:
1. `clean_data_file()` normalises to `\N` format
2. Stream Load URL: `/api/{db}/{table}/_stream_load`
3. Headers include column list, separator, null format, unique label (UUID), and 7,200-second timeout
4. Handles redirects and validates response JSON

### 5.3 Trailing Pipe Handling

Both `dbgen` (TPC-H) and `dsdgen` (TPC-DS) produce data files where each line ends with a trailing pipe delimiter. This extra field must be handled during loading to avoid "extra column" errors:

| Engine | Approach |
|:-------|:---------|
| PostgreSQL | Strip trailing `\|` from each line in Python before feeding to COPY |
| DuckDB | Explicit column selection in `read_csv()` / `COPY` to ignore extra field |
| CedarDB | `clean_data_file()` produces normalised temporary file |
| StarRocks | `clean_data_file()` produces normalised temporary file |

### 5.4 Combined Loading for Benchmarking

When running the unified benchmark via `benchmark.py`, both TPC-H and TPC-DS data are loaded into the same database instance. The `load_all_data()` function orchestrates:

1. Clean isolation schemas from any previous run (`_cleanup_isolation_schemas()`)
2. Load TPC-H data via `TPC-H/run.sh load`
3. Isolate TPC-H customer table → `tpch.customer` (PostgreSQL/CedarDB only)
4. Prepare TPC-DS schema via `TPC-DS/run.sh prepare`
5. Load TPC-DS data via `TPC-DS/run.sh load`
6. Isolate TPC-DS customer table → `tpcds.customer` (PostgreSQL/CedarDB only)

This avoids the collision between the TPC-H `customer` table (5 columns) and TPC-DS `customer` table (18 columns), which share the same unqualified name.
